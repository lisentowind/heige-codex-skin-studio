# 主题中心版本检查与 Codex 更新指令设计

## 一、目标

在 HeiGe 主题中心内显示当前安装版本。用户主动点击后检查 GitHub 最新正式 Release；发现新版本时，可一键复制一段完整更新指令并粘贴给 Codex 执行。

本功能保持轻量，不自动联网、不后台轮询、不自行下载或覆盖安装文件，也不调用未经验证的 Codex 内部新建任务接口。

## 二、用户界面

在主题中心的「当前主题」横幅下方增加一条紧凑版本栏。

初始状态：

```text
当前版本 v5.2.2    [检查更新]
```

检查中：

```text
正在检查 GitHub…    [检查中]
```

已是最新版：

```text
v5.2.2 已是最新版    [已是最新版]
```

发现新版：

```text
发现新版本 v5.3.0    [复制更新指令]
```

检查失败：

```text
暂时无法检查更新    [重新检查]
```

复制成功后，通过现有面板提示区显示「更新指令已复制，请粘贴到 Codex 对话中执行」。复制失败时显示可理解的失败信息，不影响主题选择和皮肤常驻控制。

## 三、版本来源

当前版本必须来自已安装运行包的 `package.json`，由 Node 侧读取、校验并注入菜单数据。UI 不维护第二份手写版本常量。

版本格式只接受不带前导零的稳定三段 SemVer，例如 `5.2.2`。GitHub Release 标签只接受对应的 `v5.2.2` 形式。

最新版来自：

```text
https://api.github.com/repos/HeiGeAi/heige-codex-skin-studio/releases/latest
```

该接口只返回最新正式 Release，不采用草稿或预发布版本。响应中的 `tag_name`、`html_url` 和仓库身份必须经过严格校验。

## 四、请求架构

Codex renderer 的 CSP 可能阻止直接访问 GitHub或本机控制端点，因此版本检查复用现有 renderer 请求排队与后台控制器巡检机制。

1. 用户点击「检查更新」。
2. 菜单生成一次性 `check-update` 请求，包含请求 ID、控制能力值和当前菜单 generation。
3. 控制器下一次健康巡检读取该请求。
4. 控制器调用独立的 GitHub Release 检查模块。
5. 检查模块设置 3 秒请求超时、32 KiB 响应上限、固定 GitHub API 地址和安全请求头。
6. 控制器通过 CDP 调用当前菜单 generation 的只读回调，把校验后的结果返回请求来源 renderer。
7. 菜单更新版本栏，不重新注入主题、不关闭主题中心、不修改持久化状态。

控制器只允许一个正在进行的版本检查。同一进程内最近一次成功结果缓存 60 秒，用于合并用户的连续点击；缓存不写磁盘，重启后自动消失。

## 五、更新指令

只有检测到严格高于当前版本的正式 Release 时，才显示「复制更新指令」。

复制内容：

```text
请把 HeiGe Codex Skin Studio 更新到最新正式版：
https://github.com/HeiGeAi/heige-codex-skin-studio

当前安装版本：v{currentVersion}
检测到最新版本：v{latestVersion}

请先阅读仓库 README 和最新 Release，使用项目提供的安装方式更新。保留现有主题与用户配置，更新后验证版本号、主题切换、原生界面恢复和皮肤常驻状态。不要修改 Codex 的 app.asar。
```

复制优先使用 `navigator.clipboard.writeText`。不可用或失败时，回退到临时 `textarea` 与 `document.execCommand("copy")`。两种路径都失败时，不得显示复制成功。

## 六、安全与失败处理

1. 不接受 renderer 提供的 GitHub 地址、当前版本或最新版。
2. GitHub 请求固定为 HTTPS 官方 API 地址，禁止重定向到其他主机。
3. 不读取或发送 GitHub Token、Codex 凭据、用户文件和主题内容。
4. GitHub 超时、离线、限流、非 200 响应、超大响应或无效 JSON 均返回同一类安全失败结果，并保留重试按钮。
5. 版本检查不进入主题状态事务，不增加 revision，不触发主题修复或 Codex 重启。
6. 旧菜单 generation 的迟到结果必须丢弃，不能改变新注入菜单。
7. 检查按钮在请求进行中禁用，防止重复请求。

## 七、模块边界

新增独立 Node 模块负责：

- 读取和校验当前包版本；
- 请求并校验 GitHub 最新正式 Release；
- 比较三段 SemVer；
- 生成经过边界检查的结果对象。

主题菜单只负责：

- 渲染版本栏；
- 发出一次性检查请求；
- 接收严格结果；
- 更新按钮状态；
- 生成并复制更新指令。

控制器只负责：

- 从 renderer 提取 `check-update` 请求；
- 调用版本检查模块；
- 把结果定向返回请求 renderer。

## 八、测试与验收

### 单元测试

1. 当前版本从 `package.json` 读取且格式严格。
2. SemVer 比较覆盖相等、升级、降级和无效输入。
3. GitHub 正常响应、超时、限流、非 200、超大响应、非法标签和错误仓库 URL。
4. 菜单初始显示当前版本且不会自动发请求。
5. 点击后只排队一个 `check-update` 请求。
6. 最新版时按钮禁用且不提供更新指令。
7. 新版时生成包含项目地址和两个版本号的完整指令。
8. 剪贴板主路径、回退路径和双失败路径。
9. 旧 generation 返回结果被拒绝。
10. 版本检查不改变主题、持久化状态或 revision。

### 本机验收

1. 打开主题中心即可看到当前版本。
2. 未点击前没有 GitHub 网络请求。
3. 点击检查后在可接受时间内显示 GitHub 最新正式版本。
4. 当前版本等于最新版时显示「已是最新版」。
5. 使用测试注入模拟更高版本时出现「复制更新指令」。
6. 复制内容可以粘贴到 Codex，项目地址和版本信息完整。
7. 检查过程不关闭面板、不切换主题、不造成 Codex 卡顿。

## 九、不在本次范围

- 皮肤工具自行下载、解压、安装或重启 Codex；
- 自动创建 Codex 任务；
- 后台定时检查、红点提醒或系统通知；
- 检查预发布版本；
- Windows 真机网络行为扩展。
