# 全预设原生浅色表面兼容设计

## 目标

逐一复核 12 套内置主题在 Codex 原生界面、主题中心和阅读增强模式下的显示效果，并修复深色主题在原生白色看板上出现浅色文字的问题。修复必须是公共兼容层，不得为每套主题复制 CSS，也不得增加运行时监听器、DOM、依赖或模糊效果。

## 根因

当前皮肤把主题正文色写入根级文字 token。深色主题为了保证背景上的正文可读，正文色通常接近白色。Codex 的「环境信息」面板和顶部「打开位置」按钮仍使用原生白色背景，但其中部分文字继承根级主题正文色，因此形成浅色文字叠加白色背景。

这个问题不是主题图片、焦点或单套配色造成的。把所有深色主题改成浅色正文虽然能掩盖问题，但会破坏对话区、侧栏和阅读增强区域的对比度。

## 采用方案

在公共皮肤 CSS 中增加一个固定的原生浅色表面墨色变量，并仅覆盖两个经过真机确认的稳定区域：

1. `[data-pip-obstacle="thread-summary-panel"]` 内的按钮、截断文本和前景 token。
2. 顶部无拖拽操作区内使用 `bg-token-bg-fog` 的按钮。

固定墨色为 `#172033`。不覆盖状态图标、增删统计颜色、背景、边框或主题中心，因此不会抹掉绿色、红色等语义颜色。

## 性能约束

1. 只新增静态 CSS 变量和有限后代选择器。
2. 不新增脚本轮询、MutationObserver、事件监听器、网络请求或 DOM 节点。
3. 不新增 `backdrop-filter`，阅读增强仍不使用高斯模糊。
4. 不修改图片体积和加载路径。

## 验证范围

1. 用失败测试锁定选择器、墨色变量和语义颜色保护边界。
2. 运行皮肤 CSS、阅读增强、主题清单和全量测试。
3. 在本机依次应用 12 套内置主题，确认主题 ID、阅读增强状态和控制器健康。
4. 对深色主题检查「环境信息」面板和「打开位置」按钮的计算颜色。
5. 检查主题中心横幅、主题卡片、字体、底部开关和预览焦点。
6. 重新构建确定性 `.skill`，再走 GitHub 的 Node、macOS、Windows 和可复现打包门禁。
